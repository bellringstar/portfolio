About Me

안녕하세요, 변화의 중심에 있고 싶은 백엔드 개발자 김현종입니다. 저는 기술을 통해 사람들의 일상을 더 편리하게 만드는 것을 목표로 합니다.

객체지향 설계에 대해 깊은 관심을 가지고 있으며, 이를 바탕으로 효율적이고 확장 가능한 서버 시스템을 구축하는 데 주력하고 있습니다. 새로운 기술을 학습하고 적용하는 것을 즐기며, 특히 대규모 데이터 처리 및 동시성에 관심이 많습니다.

팀과의 협업을 통해 더 나은 솔루션을 만들어내는 과정에서 큰 보람을 느낍니다. 효과적인 의사소통과 지식 공유를 통해 팀 전체의 성장에 기여하고자 노력합니다.

저는 다음과 같은 가치를 중요시합니다:

첫째, "효과적인 의사소통"과 "균형 잡힌 팀워크"입니다. 우아한테크캠프에서 진행한 `축제의 민족` 프로젝트에서는 제한된 리소스 환경에서의 대규모 트래픽 처리라는 도전적인 과제를 맡았습니다. 이 과정에서 단순히 기술적 해결책을 제시하는 것을 넘어, 팀원들과 함께 시스템의 한계점을 분석하고 개선 방향을 도출했습니다. 특히 결제 시스템 설계 시에는 팀원들과의 적극적인 토론을 통해 비동기 처리 방식을 채택했고, 이는 시스템 성능 향상으로 이어졌습니다.
둘째, "지속적인 학습과 기술적 도전"입니다. 현재는 대규모 트래픽 처리와 시스템 안정성 향상에 집중하고 있습니다. 매주 진행하는 스터디와 함께, 동시성 제어에 대한 심도 있는 학습을 진행하고 있습니다.
셋째, "문제 해결과 개선에 대한 끊임없는 고민"입니다. 모든 프로젝트에서 단순히 기능 구현에 그치지 않고, 시스템의 안정성과 확장성을 고려한 설계를 지향합니다. 현재는 대용량 데이터 처리와 트랜잭션 관리에 대한 이해를 높이는 데 집중하고 있습니다.

사용 기술

Java 17 Spring Boot 3.3.2 MySQL 8.0.35 Redis React K6 Prometheus Grafana AWS

담당 업무

  • 결제 비동기 처리 시스템 구현
    • 배경
      • 동시 결제 요청 시 DB 커넥션 풀 고갈로 인한 시스템 불안정 발생
      • 제한된 리소스(t3.small) 환경에서의 처리량 한계
      • 결제 처리 지연으로 인한 사용자 경험 저하
    • 고민 과정
      • 동기식 처리 방식 분석
        • DB 커넥션 풀 사이즈 증가: 리소스 제약으로 인해 현실적이지 않음
        • DB 샤딩: 시스템 복잡도 증가 및 운영 부담 증가
      • 메시지 큐 도입 검토
        • Kafka 도입: 추가 인프라 비용 및 운영 복잡도 증가
        • Redis Pub/Sub: 메시지 유실 가능성과 영속성 부재, 레디스 부담 증가
      • 비동기 처리 방식 선택 이유
        • 추가 인프라 없이 구현 가능한 경량화된 솔루션
        • Java의 비동기 프로그래밍 지원 활용 가능
        • 기존 시스템 아키텍처 유지하며 성능 개선 가능
    • 구현 내용
      • CompletableFuture 기반 비동기 처리
        • 커스텀 스레드 풀 구성: CPU 코어 수 기반 optimal 스레드 수 설정
        • 비동기 작업 체이닝으로 결제-주문-알림 플로우 구성
        • 예외 처리를 위한 exceptionally 핸들러 구현
      • 결제 상태 관리
        • Caffeine 캐시 도입으로 DB 조회 부하 감소
        • 캐시 정책: 5분 TTL, 최대 10000개 엔트리
        • 캐시 무효화 전략: Write-Through 방식 적용
  • MySQL 배치 처리 최적화
    • 배경
      • 개별 결제건마다 DB 연산 수행으로 인한 커넥션 부족 현상 발생
      • 대량의 주문 데이터 처리 시 성능 저하 문제
      • 티켓팅 오픈 직후 급증하는 트랜잭션 처리 필요
    • 고민 과정
      • 개별 처리 방식 분석
        • 커넥션 풀 설정 조정: 서버 리소스 한계로 근본적 해결 어려움
        • 트랜잭션 타임아웃 조정: 사용자 경험 저하 우려
      • 데이터 저장소 변경 검토
        • Redis 활용: 속도는 개선되나 데이터 정합성 이슈 우려
        • MongoDB 도입: 스키마 변경 비용과 학습 곡선 고려
      • 배치 처리 방식 선택 이유
        • 단일 트랜잭션으로 다수의 데이터 처리 가능
        • DB 커넥션 사용 효율성 극대화
        • 기존 RDBMS 활용으로 안정성 확보
    • 구현 내용
      • 인메모리 큐 시스템 구축
        • ConcurrentLinkedQueue 활용한 스레드 세이프한 데이터 적재
        • 큐 사이즈 모니터링 및 임계치 기반 배치 처리 트리거 구현
        • 주기적 배치 처리를 위한 스케줄러 구현
      • 배치 INSERT 최적화
        • JdbcTemplate batchUpdate 활용한 벌크 인서트 구현
        • 배치 사이즈 실험을 통한 최적 처리량 도출 (500건 단위)
        • rewriteBatchedStatements=true 설정으로 MySQL 배치 성능 최적화
      • 장애 대응 로직
        • 배치 실패 시 개별 건 재처리 메커니즘 구현
        • 처리 실패 데이터 로깅 및 복구 프로세스 수립
    • 결과 및 개선 고려사항
      • 성과
        • DB 커넥션 사용량 70% 감소
      • 향후 개선 방향
        • 현재 인메모리 큐 사용으로 인한 데이터 유실 가능성 존재
        • Kafka 도입을 통한 메시지 영속성 확보 방안 검토
        • 배치 사이즈 동적 조정 메커니즘 도입 고려
  • 장애 대응 메커니즘 구현
    • 배경
      • 결제 실패나 시스템 장애 시 데이터 정합성 보장 필요
      • 배치 처리 실패 시 데이터 복구 방안 부재
      • 장애 상황에서의 사용자 경험 개선 필요
    • 고민 과정
      • 분산 트랜잭션 패턴 검토
        • 2PC(Two-Phase Commit): 성능 오버헤드와 복잡도 증가
        • SAGA 패턴: 구현 복잡도와 운영 부담 고려
        • 보상 트랜잭션: 구현 용이성과 안정성 확보 가능
      • 장애 복구 전략
        • 자동 롤백: 예측하지 못한 상황 발생 가능성
        • 수동 개입: 운영자의 즉각적인 대응 어려움
        • 하이브리드 방식: 자동화된 복구와 수동 개입 결합
    • 구현 내용
      • 보상 트랜잭션 처리
        • 결제-주문-재고 상태 동기화를 위한 보상 로직 구현
        • 실패 이벤트 로깅 및 모니터링 시스템 연동
        • 운영자 알림 시스템 구축
      • 배치 작업 재처리
        • 재처리 횟수 제한 및 점진적 백오프 전략 적용
        • 수동 개입이 필요한 케이스 분류 및 대응 프로세스 수립
  • 프론트엔드 화면 구성
    • React를 활용한 사용자 인터페이스 개발
